        \item 工具先输出 10 条结构化风险项，\texttt{REQ-POSE-001} 初始 \texttt{Impact=4}、\texttt{Risk Score=12}

\begin{frame}{风险驱动测试策略}
    \small
    \begin{columns}[T]
        \column{0.52\textwidth}
        \begin{block}{高优先级风险}
            \begin{itemize}
                \item 越权访问：\texttt{/api/user/\{userId\}} 相关接口缺少鉴权
                \item 开放跨域：\texttt{origins="*"} 带来的调用边界风险
                \item 高频姿态请求：实时分析场景下潜在性能瓶颈
                \item 仪表板统计：\texttt{duration} 字段语义与聚合口径需要核对
            \end{itemize}
        \end{block}

        \column{0.42\textwidth}
        \footnotesize
        \begin{tabular}{@{}ll@{}}
            \toprule
            \textbf{优先级} & \textbf{对应套件} \\ \midrule
            P1 & 安全测试 \\
            P2 & 姿态分析 EP/BVA/ST \\
            P3 & 记录过滤 DT/WBJ \\
            P4 & 仪表板与历史记录 \\
            P5 & 回归与扩展验证 \\
            \bottomrule
        \end{tabular}
    \end{columns}

    \vspace{0.5em}
    \footnotesize
    基本思路是先用 QRA 找出高风险路径，再把黑盒、白盒与自动化执行资源优先投入到这些路径上。
\end{frame}

\begin{frame}{关键设计实例}
    \small
    \begin{block}{边界值分析实例}
        \begin{itemize}
            \item 最初工具会把 \texttt{32/33/34} 视为对称边界，但真实实现不是严格的“只允许 33 点”
            \item 最终经执行验证，\texttt{32} 点触发 \texttt{4xx}，而 \texttt{34/35} 点仍可兼容解析
            \item 这个案例说明人工审查对 AI 生成结果是必要的
        \end{itemize}
    \end{block}

    \begin{block}{决策表与组合测试实例}
        \begin{itemize}
            \item 决策表覆盖 \texttt{count < 3 \&\& duration < 30} 的四种真假组合
            \item Pairwise 不再停留在抽象 difficulty/skipRest，而是落到真实的历史记录筛选参数
            \item 这样生成的测试更贴近现有 API 和自动化脚本
        \end{itemize}
    \end{block}
\end{frame}

\begin{frame}{工具价值与局限}
    \small
    \begin{columns}[T]
        \column{0.48\textwidth}
        \begin{block}{工具价值}
            \begin{itemize}
                \item 将需求、风险、覆盖项、用例和执行证据串起来
                \item 支持交互审查，避免直接照搬 AI 输出
                \item 对 API/业务规则型系统尤其有效
            \end{itemize}
        \end{block}

        \column{0.48\textwidth}
        \begin{block}{当前局限}
            \begin{itemize}
                \item UI 层与浏览器行为未作为本轮重点
                \item 安全与性能测试仍以补充验证为主
                \item 部分语义仍需依赖人工结合代码和执行结果校正
            \end{itemize}
        \end{block}
    \end{columns}

    \vspace{0.5em}
    \footnotesize
    因而本项目最终采用“工具生成 + 人工审查 + 自动化执行证据”三段式验证策略。
\end{frame}
